# Agent Commerce 最小可行闭环与协议拆解

---

## 一、 场景

### 1. 链上多智能体量化基金自主采购第三方加密 Alpha 策略信号流

在量化交易系统（如多智能体量化架构）的日常运行中，调度 Agent 需要实时向外部的独立策略 Agent 采购高价值的、受隐私保护的交易信号。该场景不仅涉及资产转移，还高度依赖于前置的风控隔离和后置的链上交付验证，是典型的 Agent Commerce 闭环。

### 2. 五大核心角色拆解
*   **下单人 (Orderer) —— 基金调度 Agent (DispatchAgent)：** 负责监控市场波动。当发现特定标的流动性突变时，触发采购需求，向外部寻找匹配的空头对冲 Alpha 信号。
*   **执行人 (Executor) —— 策略生产 Agent (AlphaAgent)：** 链下运行的高性能模型，负责实时跑数据并生成加密的交易信号数据包（包含目标标的、开仓方向、时间戳）。
*   **验收人 (Acceptor) —— 链上零知识验证器合约 (ZK-Evaluator Contract)：** 负责对交付的信号哈希及时间戳进行无信任验证，确保该信号未超时、排除了未来函数，且格式符合技术规范。
*   **付款人 (Payer) —— Cobo CAW (Agentic Wallet) 托管账户：** 量化基金的链上资产金库。由人类主管提前为其配置好底层风控策略。
*   **仲裁人 (Arbitrator) —— 去中心化仲裁协议 (如 Kleros) / 预设多签网关：** 当交付物格式正确但验证码失效，或发生链上不可抗力拥堵时，介入处理资金流向。

## 二、 最小商业与支付流程 (Minimum Commerce Flow)

整个链路拒绝传统的“盲目授权”模式，引入 **x402** 作为机器账单协议，并依靠 **Cobo CAW 的 Pact 机制** 进行基础设施级别的风控拦截，流程图及详细步骤如下：

    [1. 发现与报价] ──> [2. 预算授权 (Pact 锁定)] ──> [3. 触发 x402 账单]
                                                               │
    [6. 审计结单] <── [5. 链上验收与结算 (Pact 解锁)] <── [4. 执行与交付]
                            │ (超时/校验失败)
                            └──> [5b. 自动退款解冻]

### 1. 报价 (Quotation)
外部的 `AlphaAgent` 在链上公告牌或特定 P2P 市场上挂牌其 API 接口：*“单次高频 Alpha 信号调用，收费 10 USDC”*。

### 2. 预算授权 (Budget Authorization)
`DispatchAgent` 决定发起采购。此时，它无法直接调用基金金库的私钥，而是通过金库的 Cobo CAW 发起一个专属的 **Pact（契约）**：
*   **Intent (意图):** 仅限于向 `AlphaAgent` 的指定合约采购交易信号。
*   **Policies (策略):** 单次调用最高预算 10 USDC，限制只能调用特定的结算 ABI 接口。
*   **Completion Conditions (完成条件):** 交互成功或时间窗口过去 5 分钟后，Pact 权限自动撤销。
> **风控意义：** 无论大模型在此期间是否遭遇 Prompt 注入攻击，其在基础设施层的操作边界已被 Pact 锁死，无法单方面转走金库内的其他清算资金。

### 3. 执行 (Execution)
`DispatchAgent` 向服务商发起 API 请求，服务端立刻返回 `HTTP 402 Payment Required` 状态码，并附带符合 **x402 协议** 规范的明细（包含接收地址、金额 10 USDC、代币类型）。

### 4. 交付 (Delivery)
消费端 Agent 识别出 x402 账单，在 Pact 的预算边界内完成链上临时托管仓（Escrow）的资金锁死。服务商 `AlphaAgent` 监测到资金锁定凭证后，立刻执行计算，并将加密的 Alpha 信号数据包及对应的 ZK-Proof 交付给验证通道。

### 5. 验收与付款 / 退款 (Acceptance & Settlement)
*   **正常结算：** `ZK-Evaluator` 智能合约自动调用验证接口。一旦证明通过，触发 Pact 的完成条件，托管仓的 10 USDC 自动释放给服务商，消费端 Agent 获得解密密钥，成功提取信号流。
*   **失败处理与争议：** 若服务商超时未交付，或 ZK 校验未通过，Pact 到期自动关闭，托管资金原路解冻返回基金金库，整个过程无需人工干预。

### 6. 记录证明 (Audit Record)
全链路的动作（Pact 的创建与撤销日志、x402 的账单哈希、链上验证器的 Success 凭证）将被永久写入区块链及 Cobo CAW 的审计日志。这为量化基金后续的财务审计、策略胜率复盘提供了不可篡改的链上证据链。

## 三、 核心协议深度对比 (x402 vs ERC-8183)

在 Agent 经济的完整生命周期中，没有一个协议能解决所有问题。以下是 **x402**（网关层）与 **ERC-8183**（业务托管层）的对比与协同：

| 比较维度 | x402 (Machine Payment Framing) | ERC-8183 (Agent Trust & Job Escrow) |
| :--- | :--- | :--- |
| **所处生命周期** | **支付触发与交互层（Checkout / Invoice）** | **业务逻辑与资产托管层（Job / Escrow / Dispute）** |
| **核心解决的环节** | **解决“机器如何看懂账单”的问题。** | **解决“机器之间如何安全履约”的问题。** |
| **机制与实现** | 扩展传统的 Web2 HTTP 状态码，将链上支付条件（地址、金额、链 ID）结构化序列化到 Header 或 Body 中，让 Agent 能够自动解析付费墙。 | 在以太坊智能合约层，定义了标准的 Job 状态、托管资金池（Escrow）和第三方验收人（Evaluator）的规范接口。 |
| **局限性** | 它只管“发起账单并提示怎么付”，但不管付了钱之后对方给不给货，不具备链上防赖账的托管仲裁能力。 | 它只管“链上资金按逻辑解锁”，但缺乏一套能够跨越 Web2/Web3 边界、让 Agent 在发起 HTTP 请求时就直接读懂的网关层协议。 |

### 协议协同关系
在实际的商业链路中，这两者是**完美的前后端互补关系**：
1.  **前端网关（x402）：** Agent 通过发送 API 请求触发 `x402`，自动获取了标准化账单。
2.  **后端安全网（ERC-8183）：** Agent 拿着账单，并不直接转账给对方，而是依照 `ERC-8183` 标准，将资金打入一个链上托管合约，并挂钩一个状态机。等服务商交付成果并通过验证后，由托管合约释放资金。
