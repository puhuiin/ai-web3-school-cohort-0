# Week 2 | Agent Identity | Agent Profile 与能力声明草图

## 研究对象

当前全球领先的 Web3 智能合约审计公司 **CertiK** 及其核心产品 **Skynet（天网）** 的业务逻辑反向工程与抽象，设计了一个“Web3 实时安全监控 Agent”的 Profile。

---

## 一、 Agent 核心定义与工作流 (Core Definition & Workflow)

### 选择的 Workflow/Agent**：CertiK Skynet Sentinel (实时智能合约安全与风险评估 Agent)


### 工作流剖析

* **Capability (核心能力)**：
* **静态代码级扫描**：自动化识别基础漏洞（如重入攻击、权限控制缺失）。
* **链上异常资金流监控**：实时追踪大额资金转移、闪电贷攻击预警。
* **链下社交情绪分析**：监控 Twitter、Discord 等社交媒体，评估项目方的“跑路 (Rug Pull)” 风险和社区情绪。


* **Input (输入)**：
* 智能合约的地址及部署网络。
* 项目方的官方社交媒体账号（Twitter、GitHub 链接）。
* 实时发生的链上交易哈希（TxHash）。


* **Output (输出)**：
* 动态更新的 **Security Score (0-100 安全评分)**。
* 实时的风险预警推送（Webhook/Telegram 报警机制）。
* 结构化的漏洞态势感知数据板（Dashboard Data JSON）。


* **Collaborators (协作对象)**：
* **交易所风控 Agent**：例如币安 (Binance) 的自动充提 Agent，在收到 Skynet 极高风险警报时，自动暂停相关代币的充提。
* **人类审计专家 (Human Auditors)**：处理 Agent 无法自行判断的深层次逻辑（如复杂的数学模型）。
* **普通 Web3 投资者**：通过前端网页查看 Agent 输出的评分，辅助投资决策。


* **Failure Points (失败点)**：
* **零日漏洞 (Zero-day Exploits)**：遭遇全新的黑客攻击手法，Agent 的历史模型无法识别，导致“漏报”。
* **链上 RPC 节点延迟/宕机**：由于区块链网络极度拥堵，Agent 无法及时拉取最新区块数据，导致“报警延迟”。
* **代码极度混淆**：无法通过自动化引擎还原真实的业务逻辑。

| **由谁维护？(Maintainer)** | 由 **CertiK 官方安全研发团队** 集中维护。他们负责定期更新 AI 扫描模型权重、CVE 漏洞库以及威胁情报源。 |
| **能做什么？(What)** | 360 度无死角评估 Web3 项目安全：从代码静态漏洞、链上资金异动、团队 KYC 状态到社区活跃度，提供综合“健康度评级”。 |
| **如何被调用？(Interface)** | 1. **B2B 机构级调用**：通过 RESTful API / GraphQL 接入交易所或钱包底层的风险拦截系统。<br>

<br>2. **B2C 社区级调用**：用户通过访问 Skynet 网页端数据看板，直接查看其工作结果。 |
| **如何收费？(Monetization)** | 采用 **SaaS 订阅制 + API 按量计费 (Pay-as-you-go)**：<br>

<br>- 基础安全看板对普通散户**免费**（作为获客和展示实力的手段）。<br>

<br>- 对交易所、项目方和机构投资者，按照 API 调用的频次和深度（例如：每 10,000 次深度链上数据查询收费 $X）进行月度结算。 |
| **如何被验证？(Verification)** | **中心化背书 + 链上存证**：<br>

<br>其信任基础主要来源于 CertiK 在行业内的权威品牌声誉。同时，Agent 生成的重要审计报告哈希值（Hash）会被锚定记录到区块链上，确保历史审查结果“不可篡改、不可抵赖”。 |
| **失败如何处理？(Fallback)** | **人机协同降级机制 (Human-in-the-loop)**：<br>

<br>如果 Agent 在代码扫描中遇到置信度低于 70% 的可疑逻辑，或者监控到难以定性的巨额资金转移，Agent 会自动挂起该任务，并向 CertiK 内部的资深人类安全工程师发送工单，由人工接管审查。 |

## 三、 协议对比分析 (MCP vs. ERC-8004)

Agent彻底走向“去中心化”与“自动化协作”，我们需要解决两大问题：**如何理解上下文环境**，以及**如何在机器间建立信任**。

### 1. MCP (Model Context Protocol)

* **核心定位**：解决 Agent **“如何读取外部工具与数据上下文”** 的接口问题。
* **适合解决的问题**：适合解决 **Agent 的输入源碎片化**问题。
* **CertiK 场景应用**：目前的 Skynet 需要为其监控的每一个代码库、每一个区块链浏览器编写不同的 API 爬虫。如果未来全网普及 MCP 协议，CertiK Agent 就可以通过统一的标准接口，直接无缝读取任何 Web3 开发者本地 IDE 里的代码或 GitHub 私有库中的上下文，极大降低了数据接入的摩擦。

### 2. ERC-8004 (Trustless Agents Standard)

* **核心定位**：解决 Agent 与 Agent 之间 **“身份认证、资质验证与声誉结算”** 的信任问题。
* **适合解决的问题**：适合解决在无人类干预的网络中，**防范女巫攻击和建立机器间信任**的问题。
* **CertiK 场景应用**：这是对安全 Agent 最致命、最关键的协议。未来，当一个完全自动化的 DeFi 部署 Agent 想要发币时，它如何知道对面的“安全 Agent”真的是 CertiK 官方的，而不是黑客伪造的？通过 ERC-8004，CertiK 可以为自己的 Skynet Agent 铸造不可篡改的链上身份护照。上游 Agent 只需要查询该护照上的“历史排雷成功率（声誉分）”，就能放心地将代码交给它审计，实现 **A2A（Agent-to-Agent）的无需信任协作**。

**MCP** 是 Agent 的“五官和感官”，决定了它能多顺畅地获取审计资料；而 **ERC-8004** 是 Agent 的“身份证和执业资格证”，决定了它在去中心化网络中能否被其他 Agent 信任并雇佣。
