## 二、 核心方向深度辨析 (Why AI × Web3?)

感兴趣的两个核心方向：

###  1：Wallet / Permission / Safe Execution (钱包与安全执行)

纯AI是概率性系统，存在幻觉，且无法自我强制约束。AI可以写出完美的交易代码，但它无法持有私钥，更无法在没有外部强制规则（如预算上限、合约白名单）的情况下，保证资金的绝对安全。
* **为什么它不是纯 Web3 问题？**
纯 Web3 钱包（如 Metamask 或传统的 Safe 多签）需要人类手动拼接交易、逐字确认合约地址并签名。它极其安全，但也极其僵化，缺乏语义理解能力，无法实现“基于模糊意图的自动化执行”。Web3 缺的是一个聪明的“大脑”。

###  2：Payment / Commerce / Settlement (支付与商业结算)

AI Agent 可以自动爬取数据或生成代码，但它在传统金融体系（如 Stripe/PayPal）中是“黑户”，无法通过 KYC 拥有银行账户，因此无法自主购买 API 或为自己的服务收费。没有 Web3，Agent 就只能是个玩具，无法形成商业闭环。
* **为什么它不是纯 Web3 问题？**
加密货币和智能合约提供了完美的支付轨道，但轨道本身不会主动去买东西。纯 Web3 缺乏对非结构化商业场景的理解（例如：如何判断一份研究报告是否符合质量要求并触发付款？）。Web3 需要 AI 来充当“消费者”和“验收员”。

# Wallet / Permission / Safe Execution 完整学习笔记

### 核心认知：为什么 Agent 需要专属的钱包架构？

传统的 Web3 钱包（如 MetaMask EOA 账户）只有两种状态：**“拥有私钥 = 拥有一切权限”** 或 **“没有私钥 = 什么都干不了”**。
如果你把私钥直接交给大模型（Agent），一旦遭遇 Prompt Injection（提示词注入）攻击或模型幻觉，你的资产瞬间归零。

因此，Agent 需要的是一套**可编程、可分层、可撤销**的智能钱包体系。

---

###  核心技术

Agent 钱包五个核心概念：

#### 1. 账户抽象 (Account Abstraction / ERC-4337)

* **本质：** 将“签名权（Signer）”和“账户资产（Account）”解耦。你的账户变成了一个部署在链上的**智能合约 (Smart Contract Account, SCA)**。
* **对 Agent 的意义：** 既然账户是智能合约，我们就可以把“必须人脸识别”、“单日限额 1000 U”、“只能买 ETH”等逻辑**写进代码里**，从底层硬性约束 Agent 的行为。

#### 2. Safe (前身为 Gnosis Safe)

* **本质：** 业界最成熟的多签智能合约钱包标准。
* **对 Agent 的意义：** 它是目前 Agent 接触大额资产的最佳载体。Agent 可以作为 Safe 的“多签人”之一，或者通过 Safe 的插件系统（Modules）获得受限的执行权。

#### 3. 会话密钥 (Session Key)

* **本质：** 一把“一次性、有时间限制、有范围限制”的临时钥匙。
* **使用场景：** 你不需要把钱包主控权给 Agent，而是给它发一张“Session Key”。
* *例子：* 授权 Agent 在 **接下来的 24 小时内**，只能在 **Uniswap 合约** 上，最多动用 **500 USDC** 进行 Swap。时间一到或额度用完，钥匙自动作废。


#### 4. Policy (策略) & Guard (守卫)

* **Policy (策略)：** 定义规则的逻辑文本（比如“禁止与 Tornado Cash 交互”）。
* **Guard (守卫)：** Safe 钱包中的一种特殊钩子（Hook）。在 Agent 发起任何交易前，Guard 会强制检查这笔交易是否符合 Policy。**如果不符合，Guard 会在链上直接拦截（Revert）这笔交易。**

---

###  工作流：权限分层与自动化边界

设计一个可靠的 Agent 钱包系统，核心在于画好**自动化与人工的边界**。

#### 1. 权限分层 (Permission Tiering)

* **读取层 (Read-only)：** Agent 可以无限次读取链上数据、余额、模拟交易结果（无需授权）。
* **低危执行层 (Low-risk)：** Agent 在限定的 **预算 (Budget)** 和 **白名单合约 (Allowlist)** 内，自主签名并上链（如：每日领取一次空投、小额定投）。
* **高危执行层 (High-risk)：** 涉及大额转账、合约授权 (Approve)、跨链操作时，Agent 只能生成待签名的 Payload，必须进入队列等待人类确认。

#### 2. 自动化边界与人工确认 (Human-in-the-loop)

* **自动化的边界在于“确定性”。**
* 系统必须设计一个**拦截阈值**。当 Agent 遇到滑点突增、非白名单代币、或者动用资金超过单笔上限时，系统必须暂停自动化流，向用户的手机（如 Cobo 的机构钱包或 Safe 移动端）推送一条审批请求。**人工点击 Confirm，交易才会被广播。**

#### 3. 撤销 (Revocation) 与 审计 (Auditing)

* **一键撤销 (Kill Switch)：** 如果发现 Agent 行为异常，人类必须能够随时撤销 Agent 的 Session Key，甚至冻结智能账户，切断其链上执行能力。
* **可验证审计 (Audit Trail)：** Agent 的每一步操作（意图、生成的 Payload、Policy 检查结果、链上 Hash）都必须被结构化记录。出了问题，必须能通过日志复盘是模型幻觉、策略配置错误，还是遭受了外部攻击。

###  核心机制模拟：Policy Engine (策略引擎) 执行逻辑示例

为了直观理解 Policy 和 Guard 的工作原理，可以假设设定了以下拦截机制规则：

* **初始状态：** 每日授权预算 1000 USDC。
* **策略规则：**
* **白名单合约：** Uniswap, Aave。
* **黑名单合约：** Tornado Cash (高危混币器)。
* **自动执行阈值：** 单笔 $\le$ 100 USDC。


* **执行场景判定：**
* *场景 A (金额 50 USDC，目标 Uniswap)：* 满足白名单且小于阈值 $\rightarrow$ **状态：自动执行成功**，从总预算扣除 50 USDC。
* *场景 B (金额 200 USDC，目标 Aave)：* 满足白名单但大于阈值 $\rightarrow$ **状态：拦截，等待人工签名确认**，暂停自动化流。
* *场景 C (金额 50 USDC，目标 Tornado Cash)：* 命中黑名单合约 $\rightarrow$ **状态：警告，权限不足，触发安全拦截**，直接 Revert 交易。
